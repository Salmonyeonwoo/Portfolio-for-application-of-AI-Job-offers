AI 학습 코치 포트폴리오

기술적 도전과제: 지속성 계층(Persistence Layer) 구축 성공 스토리

1. 도전 과제 개요

Streamlit Cloud 환경에서 RAG 인덱스(FAISS)와 데이터를 영구적으로 저장하기 위해 Google Cloud Firestore(GCP)를 선택했으나, 예상치 못한 서버 인증 충돌 문제에 직면했습니다.

핵심 문제: Admin SDK 초기화 실패

**오류 메시지:** LLM 초기화 오류: API 키를 확인해 주세요. (DB Auth Error: Invalid \escape) / The default Firebase app does not exist.

2. 단계별 문제 해결 과정

단계 1: Streamlit 실행 순서 및 코드 정렬

가장 먼저, `st.set_page_config`가 다른 Streamlit 명령보다 먼저 실행되도록 코드 구조를 정리하고, Firebase 초기화 로직을 분리했습니다.

# Code Reordering: Configuration -> st.set_page_config -> Initialization

단계 2: Secrets JSON 문자열 복구 (Invalid \escape 해결)

Secrets에 입력된 `private_key`의 `\n` 이스케이프 문자가 잘못 해석되는 문제를 해결하기 위해, 강력한 문자열 정제 로직을 Admin SDK 로더에 삽입했습니다.

복구 로직 적용

코드: `sa_info_str.replace('\\n', '\n')` 로직 삽입

Secrets 입력 방식 (참조용)

외부 따옴표 없이 JSON 원본 입력 (Secrets UI 캡처본 대체)

단계 3: Admin SDK 캐시 충돌 해결

`The default Firebase app does not exist` 오류는 캐시 환경 충돌 문제였습니다. 초기화 로직을 캐시 함수 외부로 분리하고, DB 클라이언트 생성만 캐시 함수(`@st.cache_resource`) 내에서 호출하도록 수정했습니다.

# Final Initialization Logic:
# sa_info = _get_admin_credentials() 
# db, error = initialize_firestore_client(sa_info)

단계 4: GCP IAM 권한 최종 확보 (DB 저장 성공)

모든 코드가 정상 작동했음에도 'DB 저장 실패' 오류가 지속되었습니다. 이는 코드 오류가 아닌 GCP의 **서비스 계정 권한 부족** 문제였습니다. IAM에서 DB 쓰기 권한을 명시적으로 부여하여 문제를 해결했습니다.

  • **부여 역할:** Cloud Datastore User, Firebase Developer, Cloud Firestore 서비스 에이전트.
  • **최종 결과:** 인덱스 저장(Persistence Layer) 성공!
