Morpheus 캡스톤 다중 입력/출력 (MIMO) 파이프라인 요약

프로젝트 과정 해석 및 기술 역량 상세 설명 (코드 제외)

프로젝트 개요 및 목표

NVIDIA 교육 과정의 **캡스톤 프로젝트**로, GPU 텔레메트리(NVSMI)와 네트워크 패킷(PCAP)이라는 **두 가지 유형의 입력 데이터**를 받아, 유형에 따라 서로 다른 ML 모델로 라우팅하여 암호화폐 채굴 활동을 탐지하는 **MIMO(Multi-Input/Multi-Output) 비선형 파이프라인**을 완성했습니다.

캡스톤 프로젝트 MIMO Triton Inference

기술적 도전 및 문제 해결 과정

1. 이질적인 데이터 통합 처리 문제

**도전:** GPU 로그와 네트워크 패킷이라는 두 개의 이질적인 데이터 소스를 하나의 파이프라인에서 처리해야 했습니다.

**해결:** `Pipeline` 객체에 **두 개의 독립적인 `FileSourceStage`**를 설정하여 멀티-입력을 구현했습니다. 각 소스 뒤에 **커스텀 메타데이터 태깅 스테이지**를 배치하여 메시지에 'NVSMI' 또는 'PCAP' 태그를 부여함으로써 데이터의 정체성을 명확히 했습니다.

2. 조건부 라우팅 및 다중 추론 구현

**핵심 로직:** 태깅된 메타데이터를 기반으로 **`RouterStage`**를 사용하여 메시지를 NVSMI 전용 브랜치(XGBoost 모델 1)와 PCAP 전용 브랜치(XGBoost 모델 2)로 분기했습니다. 각 브랜치에는 **`TritonInferenceStage`**를 통해 해당 데이터에 최적화된 ML 모델 추론을 통합했습니다.

**성과:** 하나의 파이프라인에서 두 가지의 독립적인 보안 위협 탐지 로직이 병렬적으로 실행되는 **실전적인 다중 탐지 시스템**을 구현하여 Morpheus의 비선형 및 다중 입력 처리 능력을 완벽하게 입증했습니다.

3. 파이프라인 모니터링

두 브랜치의 최종 출력을 다시 하나로 모아(add_edge로 연결) **`MonitorStage`**를 배치했습니다. 이를 통해 파이프라인 전체의 통합 처리량(throughput)을 모니터링하여 시스템의 성능을 지속적으로 측정했습니다.

API 키 및 코드 처리 방안

이 프로젝트는 내부 모델 추론(Triton 서버 활용)에 집중했으며, 외부 API Key는 사용하지 않았습니다.

**코드 포메이션 대신:** **멀티-입력 및 비선형 구조 설계의 필요성**, **메타데이터 태깅을 통한 라우팅 제어 논리**, 그리고 **Triton을 통한 모델 통합 과정** 등 엔터프라이즈급 AI 시스템 설계 능력을 중심으로 기술 해석을 진행했습니다.
