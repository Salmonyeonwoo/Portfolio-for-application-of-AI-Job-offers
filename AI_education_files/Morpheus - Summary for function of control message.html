Morpheus Control Message 파이프라인 통합 및 활용 요약

프로젝트 과정 해석 및 기술 역량 상세 설명 (코드 제외)

프로젝트 개요 및 목표

Morpheus의 핵심인 `ControlMessage` 객체를 파이프라인에 통합하고, 메시지에 메타데이터와 태스크 정보(다운스트림 스테이지가 수행할 작업)를 주입하는 커스텀 스테이지(`AddMetadataAndTaskStage`)를 개발했습니다. 이는 데이터 흐름을 동적으로 제어하는 능력을 입증합니다.

ControlMessage Custom Stage 개발 동적 태스크 주입

기술적 도전 및 문제 해결 과정

1. 메시지 타입 변환 문제 해결

도전: `FileSourceStage`의 출력은 `MessageMeta`인데, `ControlMessage` 기반의 커스텀 스테이지에 연결하려면 타입 변환이 필요했습니다.

해결: 소스 스테이지 바로 뒤에 `DeserializeStage`를 배치하여 `MessageMeta`를 `ControlMessage`로 변환하도록 명시적으로 지정했습니다. 이로써 `ControlMessage`의 고급 기능을 사용할 수 있는 환경을 마련했습니다.

2. 태스크 및 메타데이터 주입 로직 개발

핵심 로직: 개발한 커스텀 스테이지(`AddMetadataAndTaskStage`)는 초기화 시점에 `metadata`와 `tasks` 딕셔너리를 인수로 받습니다. 이 스테이지의 `on_data` 메서드 내에서 메시지의 `set_metadata()`와 `add_task()` 헬퍼 함수를 호출하여 외부에서 정의된 제어 정보를 메시지에 주입하는 로직을 구현했습니다.

활용: 예를 들어, 메시지에 `{'priority': 'high'}` 메타데이터를 추가하거나, 특정 IP/Port 필터링을 지시하는 태스크를 추가하여 후속 스테이지의 처리 방식에 영향을 미치도록 했습니다.

3. 커스텀 스테이지의 타입 안정성 확보

개발된 커스텀 스테이지의 `accepted_types()` 메서드에 `(ControlMessage, )`를 명시적으로 지정하여, 이 스테이지가 오직 `ControlMessage`만 처리하도록 타입 안정성을 확보했습니다.

API 키 및 코드 처리 방안

이 프로젝트는 Morpheus 파이프라인의 제어 흐름 구현에 초점을 맞추었으며 외부 API Key를 사용하지 않았습니다.

코드 포메이션 대신: 위에 설명된 커스텀 스테이지의 역할 정의, `on_data`에서의 제어 함수 호출 로직, 메시지 타입 변환 문제 해결 과정 등 파이프라인 제어 구조 설계 능력을 보여주는 기술 해석에 집중했습니다.
